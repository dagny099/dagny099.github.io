Worked examples

These show the thought process: prompt → pattern recognition → simple code → edge cases. The goal is not memorization; it is learning how to get unstuck and move cleanly from description to implementation.

Worked example 1: first unique character

Prompt: Return the first character in a string that appears exactly once. Return None if no such character exists.

How to think about it:

  1. I care about frequency, so this suggests Counter.
  2. I also care about the first such character, so I should count first, then scan the original string in order.
  3. That means a two-pass solution.
from collections import Counter

def first_unique(s):
    if not s:
        return None

    counts = Counter(s)
    for ch in s:
        if counts[ch] == 1:
            return ch
    return None

Edge cases to check: empty string, all duplicates, uppercase/lowercase handling if the prompt cares.

Worked example 2: pair sum

Prompt: Given a list of integers and a target, return True if any two distinct numbers sum to the target; otherwise return False.

How to think about it:

  1. The brute-force approach is nested loops, but that is O(n²).
  2. For each number x, I really want to know whether target - x has already appeared.
  3. That is a fast lookup problem, so use a set.
def has_pair_sum(nums, target):
    seen = set()

    for x in nums:
        need = target - x
        if need in seen:
            return True
        seen.add(x)

    return False

Edge cases to check: empty list, one item, duplicates like [3, 3] with target 6, negatives, zero.

When you feel stuck, ask: What am I repeatedly looking up? That question often reveals the right structure.

Complement lookup / two-sum
def two_sum(nums, target):
    seen = {}
    for i, x in enumerate(nums):
        need = target - x
        if need in seen:
            return [seen[need], i]
        seen[x] = i
    return []

Sliding window
def max_sum_window(arr, k):
    if k <= 0 or k > len(arr):
        return None
    window = sum(arr[:k])
    best = window
    for i in range(k, len(arr)):
        window += arr[i] - arr[i-k]
        best = max(best, window)
    return best
Two pointers
def has_pair_sum(arr, target):
    arr.sort()
    left, right = 0, len(arr) - 1
    while left < right:
        total = arr[left] + arr[right]
        if total == target:
            return True
        elif total < target:
            left += 1
        else:
            right -= 1
    return False
Stack
def valid_parens(s):
    pairs = {")": "(", "]": "[", "}": "{"}
    stack = []
    for ch in s:
        if ch in "([{":
            stack.append(ch)
        elif ch in pairs:
            if not stack or stack[-1] != pairs[ch]:
                return False
            stack.pop()
    return not stack
Merge intervals
def merge_intervals(intervals):
    if not intervals:
        return []
    intervals.sort(key=lambda x: x[0])
    merged = [intervals[0]]
    for start, end in intervals[1:]:
        last_start, last_end = merged[-1]
        if start <= last_end:
            merged[-1] = [last_start, max(last_end, end)]
        else:
            merged.append([start, end])
    return merged

Big-O + hidden test cases

Main rule: if you wrote a nested loop over the same list, pause and ask whether a dictionary, set, Counter, or sort can replace it.

Complexity cheat sheet

PatternTypical complexityWhy it matters
One pass through listO(n)Usually good
Nested loop over same listO(n²)Dangerous for large input
SortO(n log n)Usually acceptable
Dict/set lookupAverage O(1)Great for replacing nested loops
Counter over list/stringO(n)Great for frequencies
